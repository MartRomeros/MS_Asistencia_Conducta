# Spec: Registro y visualizacion de anotaciones por docente creador

## Objetivo
Permitir que un docente registre anotaciones de conducta para estudiantes y visualice unicamente las anotaciones que el mismo creo.

La funcionalidad debe respetar el contexto actual del microservicio de asistencia y conducta, usando JWT para identificar al docente autenticado y la tabla existente `anotaciones` definida en `.agents/context/script.sql.md`.

## Alcance
- Crear endpoint para registrar una anotacion asociada a un estudiante.
- Tomar el `docente_id` desde el JWT del docente autenticado, no desde el body.
- Crear endpoint para listar anotaciones creadas por el docente autenticado.
- Permitir filtrar la visualizacion por `estudiante_id`, `tipo`, `fecha_desde` y `fecha_hasta`.
- Retornar datos utiles para UI: informacion basica del estudiante, tipo, descripcion, fecha de registro y docente creador.
- Documentar los endpoints con OpenAPI en las rutas correspondientes.
- Validar entradas con Zod.
- Mantener la arquitectura actual por capas: routes, controllers, services, models, schemas.
- Agregar tests de integracion o unitarios suficientes para los casos principales.

## Fuera de alcance
- Editar anotaciones existentes.
- Eliminar anotaciones.
- Visualizar anotaciones creadas por otros docentes.
- Crear una nueva tabla de auditoria.
- Cambiar el formato del JWT emitido por el microservicio de autenticacion.
- Implementar frontend.
- Cambiar la estructura actual de roles, usuarios, estudiantes o docentes.

## Actores
- Docente autenticado: usuario con `role = "Docente"` en el JWT.
- Estudiante: usuario existente en la tabla `estudiantes`.

## Modelo de datos existente
Tabla `anotaciones`:

```sql
CREATE TABLE "anotaciones" (
  "anotacion_id" INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  "estudiante_id" integer,
  "docente_id" integer,
  "tipo" varchar NOT NULL,
  "descripcion" text NOT NULL,
  "fecha_registro" timestamp DEFAULT (now())
);
```

Relaciones existentes:

```sql
ALTER TABLE "anotaciones" ADD FOREIGN KEY ("estudiante_id") REFERENCES "estudiantes" ("estudiante_id");
ALTER TABLE "anotaciones" ADD FOREIGN KEY ("docente_id") REFERENCES "docentes" ("docente_id");
```

## Reglas de negocio
- Solo un usuario con rol `Docente` puede registrar anotaciones.
- El `docente_id` de la anotacion debe ser siempre `payload.id` del JWT autenticado.
- El cliente no debe enviar `docente_id` en el body. Si lo envia, debe rechazarse por validacion estricta.
- El estudiante debe existir y estar activo antes de registrar la anotacion.
- Por defecto, un docente solo puede registrar anotaciones sobre estudiantes de cursos que tiene asignados en `curso_asignatura_docente`.
- Un docente solo puede listar anotaciones donde `anotaciones.docente_id = payload.id`.
- La descripcion debe tener contenido util: minimo 5 caracteres y maximo 1000 caracteres.
- El tipo debe ser un valor controlado para evitar inconsistencias. Valores iniciales propuestos: `positiva`, `negativa`, `observacion`.
- `fecha_registro` debe ser asignada por la base de datos con `now()` y no debe recibirse desde el cliente.
- El listado debe ordenarse por `fecha_registro DESC`.
- El listado debe soportar paginacion con `page` y `limit` para evitar respuestas grandes.

## Endpoints propuestos

### POST /api/anotaciones
Registra una anotacion creada por el docente autenticado.

Headers:

```http
Authorization: Bearer <jwt>
Content-Type: application/json
```

Body:

```json
{
  "estudiante_id": 104,
  "tipo": "negativa",
  "descripcion": "Interrumpe reiteradamente la clase pese a advertencias previas."
}
```

Respuesta 201:

```json
{
  "success": true,
  "message": "Anotacion registrada correctamente",
  "data": {
    "anotacion_id": 12,
    "estudiante_id": 104,
    "docente_id": 5,
    "tipo": "negativa",
    "descripcion": "Interrumpe reiteradamente la clase pese a advertencias previas.",
    "fecha_registro": "2026-05-09T23:40:00.000Z"
  }
}
```

### GET /api/anotaciones
Lista anotaciones creadas por el docente autenticado.

Headers:

```http
Authorization: Bearer <jwt>
```

Query params opcionales:
- `estudiante_id`: entero positivo.
- `tipo`: `positiva`, `negativa` u `observacion`.
- `fecha_desde`: fecha `YYYY-MM-DD`.
- `fecha_hasta`: fecha `YYYY-MM-DD`.
- `page`: entero positivo, default `1`.
- `limit`: entero positivo, default `20`, maximo `100`.

Ejemplo:

```http
GET /api/anotaciones?estudiante_id=104&tipo=negativa&page=1&limit=20
```

Respuesta 200:

```json
{
  "success": true,
  "data": [
    {
      "anotacion_id": 12,
      "estudiante_id": 104,
      "estudiante_nombre": "Juan",
      "estudiante_apellido_paterno": "Perez",
      "estudiante_apellido_materno": "Gomez",
      "docente_id": 5,
      "tipo": "negativa",
      "descripcion": "Interrumpe reiteradamente la clase pese a advertencias previas.",
      "fecha_registro": "2026-05-09T23:40:00.000Z"
    }
  ],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 1,
    "total_pages": 1
  }
}
```

## Codigos de respuesta
- `201`: anotacion creada correctamente.
- `200`: listado obtenido correctamente.
- `400`: body, params o query invalidos.
- `401`: token ausente, invalido o expirado.
- `403`: usuario autenticado no es docente o intenta operar fuera de sus permisos.
- `404`: estudiante no existe o no esta activo.
- `500`: error interno no controlado.

## Criterios de aceptacion
- Dado un docente autenticado, cuando registra una anotacion valida para un estudiante permitido, entonces se crea el registro con `docente_id` igual al ID del JWT.
- Dado un docente autenticado, cuando lista anotaciones, entonces solo recibe anotaciones creadas por ese mismo docente.
- Dado un docente autenticado, cuando filtra por `estudiante_id`, entonces solo recibe sus anotaciones para ese estudiante.
- Dado un docente autenticado, cuando filtra por `tipo`, entonces solo recibe sus anotaciones de ese tipo.
- Dado un docente autenticado, cuando filtra por rango de fechas, entonces solo recibe anotaciones dentro del rango solicitado.
- Dado un usuario sin token, cuando intenta registrar o listar anotaciones, entonces recibe `401`.
- Dado un usuario con rol distinto de `Docente`, cuando intenta registrar o listar anotaciones, entonces recibe `403`.
- Dado un docente que intenta registrar una anotacion sobre un estudiante que no pertenece a sus cursos asignados, entonces recibe `403`.
- Dado un body con `docente_id`, cuando intenta registrar una anotacion, entonces la peticion es rechazada por validacion.
- Dado un estudiante inexistente o inactivo, cuando se intenta registrar una anotacion, entonces recibe `404`.
- El build TypeScript debe pasar.
- Los tests relacionados deben pasar.

## Supuestos
- El JWT actual contiene `id`, `email` y `role`, como valida `src/services/jwt.service.ts`.
- Para docentes, `payload.id` corresponde a `docentes.docente_id` y tambien a `usuarios.usuario_id`.
- La pertenencia de un estudiante a un docente puede inferirse con `estudiantes.curso_id` y `curso_asignatura_docente.docente_id`.
- No se requiere soft delete para anotaciones en esta iteracion.
- El microservicio no es responsable de emitir el JWT, solo de validarlo.
